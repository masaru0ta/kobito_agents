# Phase 2 仕様書: タスク管理

## 概要

AIエージェントが起票するタスクを管理し、人間が実行を承認・制御する。
1タスク = 1 MDファイル。AIはタスクを起票・作業するが、実行承認は人間のみが行える。

モックアップ: [mockup_phase2.html](mockup_phase2.html)

## 設計原則

- **承認は人間のみ** — AIはメタデータファイルに書き込めない。誤実行を構造的に防止する
- **作業状態と承認状態を分離** — `phase`（AI管理）と `approval`（人間のみ）は別概念
- **MDは内容、`.kobito/`はシステム管理** — AIが読み書きするMDと、システムが管理するメタデータを分離する
- **紐づけはシステムが管理** — セッションとタスクの紐づけは、セッション開始という明示的なアクション時にAPIが書き込む
- **シンプルな一覧** — 定期タスクも通常タスクと同じリストで扱う。専用UIは持たない

## データモデル

### タスクファイル（`tasks/{task_id}.md`）

AIが作成・編集する。YAML frontmatterに作業状態を持つ。

```markdown
---
title: セッション一覧の読み込み速度改善
agent: system
phase: draft
created: 2026-04-01T17:00:00Z
---

## 背景

セッション数が100を超えると応答が2秒以上かかる。

## 作業ステップ

- [x] JONLファイルのメタ情報をキャッシュファイルに保存する
- [ ] 一覧取得時はキャッシュから返し、更新があったものだけ再パースする
```

#### frontmatterフィールド

| フィールド | 型 | 必須 | 説明 |
|-----------|-----|------|------|
| title | string | ✓ | タスクタイトル |
| agent | string | ✓ | 担当エージェントID |
| phase | string | ✓ | 作業状態: `draft` / `doing` / `done` |
| created | string | ✓ | ISO8601形式の起票日時 |
| schedule | string | — | 定期実行の設定（例: `毎週 月曜 10:00`）。指定ありで定期タスク |

IDはfrontmatterには持たない。ファイル名（拡張子を除いた部分）がタスクIDになる。

#### タスクID形式

```
task_20260401_153045_a3f
```

`task_{YYYYMMDD}_{HHmmss}_{3桁ランダム英数字}` 形式。AIがMDファイルを作成する際に自身で採番する。

#### phaseの意味

| phase | UI表示 | 説明 |
|-------|--------|------|
| `draft` | 未着手 | 起票済み・未着手 |
| `doing` | 実行中 | 作業中（● インジケーター付き） |
| `done` | 完了 | 完了 |

#### 作業ステップ

MDボディ内の `- [ ]` / `- [x]` チェックリストが作業ステップとして機能する。AIが作業完了時に `[x]` に書き換えることで進捗が更新される。タスク詳細UIで進捗バーとして可視化される。

### タスクメタデータ（`.kobito/tasks/{task_id}.json`）

サーバーのみが書き込む。AIは読み取り可能だが書き込み不可。承認状態・セッション紐づけを一元管理する。

タスク一覧取得時、対応するメタデータファイルが存在しないタスクに対してサーバーが自動生成する。

```json
{
  "task_id": "task_20260401_153045_a3f",
  "approval": "pending",
  "approved_at": null,
  "rejected_at": null,
  "reject_reason": "",
  "sessions": ["session_abc123"],
  "talk_session_id": "session_xyz789"
}
```

| フィールド | 型 | 説明 |
|-----------|-----|------|
| task_id | string | タスクID（ファイル名と一致） |
| approval | string | `approved` / `rejected` / `pending` |
| approved_at | string\|null | 承認日時（ISO8601） |
| rejected_at | string\|null | 却下日時（ISO8601） |
| reject_reason | string | 却下理由 |
| sessions | string[] | 作業セッションID一覧 |
| talk_session_id | string\|null | 相談セッションID（1タスク1セッション） |

### 実行順序（`.kobito/task_order.json`）

承認済みタスクの実行順序を管理する。サーバーのみが書き込む。

```json
["task_20260401_153045_a3f", "task_20260401_160000_b2e"]
```

- 配列インデックス = 実行優先順位（0が最優先）
- スケジューラーはこのリストの先頭タスクを次の実行対象とする
- 削除・完了したタスクIDは自動的に除去される

## タスクの状態遷移

### phaseの遷移（AIがMDを直接編集）

```
draft → doing → done
draft ────────→ done  （強制完了: 人間がWeb UIで操作）
```

### approvalの遷移（人間のみがWeb UI経由で操作）

```
(なし=pending) → approved
(なし=pending) → rejected
approved → rejected  （再却下）
rejected → approved  （再承認）
```

### 定期タスクのリセット

`schedule`フィールドがあるタスクは、`done`状態のまま次回スケジュール時刻を過ぎると自動的に `phase: draft` にリセットされる。承認状態は維持される。タイトルには `(定期)` を表示する。

## セッション紐づけ

| 種別 | フィールド | 起点 | 動作 |
|------|-----------|------|------|
| 相談セッション | `talk_session_id` | 人間が「タスクについて話す」を押す | `talk_session_id`が未設定なら新規セッション作成・保存。設定済みなら既存セッションを開く |
| 作業セッション | `sessions[]` | 人間が「1ステップ作業する」を押す / スケジューラーが自動起動 | 毎回新規セッション作成・`sessions`配列に追記 |

相談セッションは1タスクにつき1つのみ。作業セッションは複数回追加される。

## スケジューラー（未実装）

10分ごとに起動し、`task_order.json` の先頭タスクに対してAIセッションを自動開始する。

```
スケジューラー起動

GET /api/agents/{id}/tasks → order[0] を取得

```

## タスク一覧UI

3ペイン構成の中央ペイン「タスク」タブに表示する。右ペイン（セッション）はタスクタブを開いても隠れない。

### 表示構造

```
☰  CLAUDE.mdの編集UIの改善    ← 承認済みキュー（ドラッグ可）
   ● 実行中（doing時のみ）

☰  セッション一覧の読み込み速度改善

   エージェント登録APIの追加      ← 承認待ち
   [承認待ち] 2026/04/01

   ConfigManagerの実装            ← 最新完了1件（opacity 0.4）
   [完了] 2026/04/01

  過去の完了タスクを表示 (3) ▶
```

### 承認済みキュー

- `approval: approved` のタスクを `task_order.json` の順に表示
- ドラッグハンドル（`☰`）で並び替え可能。末尾ドロップゾーンあり
- `phase === 'doing'` のタスクのみ `● 実行中` インジケーターを表示

### 承認待ち

- `approval: pending` のタスクをキューの下に表示
- メタ行に `[承認待ち]` バッジを表示
- タスクタブに未承認件数バッジを表示

### 完了タスク

- 最新1件を opacity 0.4 で常時表示
- 「過去の完了タスクを表示 (N)」トグルで折りたたみ
- メタ行に `[完了]` バッジを表示

## タスク詳細UI

タスクをクリックすると**中央ペイン内**でタスク詳細に切り替わる。右ペインはそのまま維持される。

### ヘッダー

```
[←]                                                        [...]
```

- 左: 戻るボタン（一覧に戻る）
- 右: `...`メニューボタン

#### `...`メニュー項目

| 項目 | 動作 |
|------|------|
| タスクについて話す | 相談セッションを開く（既存があれば再利用、なければ新規作成） |
| 1ステップ作業する | 新規作業セッションを開始し、AIに1ステップ作業を指示 |
| 編集 | タスク本文をtextareaに切り替えてインライン編集 |
| 強制完了 | `phase: done` に強制更新（done状態では無効化） |
| 削除 | 確認ダイアログの後、MDファイルとメタデータファイルを削除 |

### ボディ構成

```
[進捗バー: ████░░  2/3]   ← チェックボックス完了数/総数（存在する場合のみ）

タスクタイトル（省略なし全文表示）

------- MDコンテンツ（Markdownレンダリング） -------

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

フェーズ: 未着手   承認待ち   起票: 2026/04/01
相談セッション: タスク名 の相談  ← talk_session_id が設定されている場合のみ
```

- 進捗バー: ステップ数の区切り線付き。チェックボックスがない場合は非表示
- 相談セッションリンク: クリックで右ペインにそのセッションを表示（タスク詳細は維持）

### 却下ダイアログ

- 理由入力テキストエリア（任意）
- `[キャンセル]` `[却下する]` ボタン

## Web API

#### タスク操作
- `GET /api/agents/{id}/tasks` — タスク一覧＋実行順序（`{tasks: [...], order: [...]}`）
- `GET /api/agents/{id}/tasks/{task_id}` — タスク詳細
- `POST /api/agents/{id}/tasks/{task_id}/approve` — 承認
- `POST /api/agents/{id}/tasks/{task_id}/reject` — 却下（ボディ: `{"reason": "..."}`）
- `POST /api/agents/{id}/tasks/{task_id}/force-done` — 強制完了
- `DELETE /api/agents/{id}/tasks/{task_id}` — タスク削除
- `PUT /api/agents/{id}/tasks/{task_id}` — タスク本文更新（ボディ: `{"body": "..."}`）
- `PUT /api/agents/{id}/tasks/order` — 実行順序更新（ボディ: `{"order": [...]}`）

#### セッション紐づけ
- `POST /api/agents/{id}/tasks/{task_id}/sessions` — 作業セッション追加（ボディ: `{"session_id": "..."}`）
- `PUT /api/agents/{id}/tasks/{task_id}/talk-session` — 相談セッションID設定（ボディ: `{"session_id": "..."}`）

#### 書き込み主体まとめ

| 操作 | 書き込み主体 |
|------|------------|
| タスク起票（MDファイル作成） | AI（直接編集） |
| タスク本文編集・phase更新 | AI（直接編集） |
| メタデータ自動生成 | サーバー（タスク一覧取得時） |
| 承認・却下・強制完了・削除 | サーバー（Web UI経由） |
| セッション紐づけ | サーバー（API経由） |
| 実行順序 | サーバー（Web UI経由） |
