# Phase 7 仕様書: チームエージェント実装

## 概要

「チーム」という特殊なエージェント種別を実装する。
チームへメッセージを送ると、LM Studio 上のファシリテーターが会話の流れに応じて
次の発言者を動的に決定し、メンバーエージェントが順番に回答する。
ファシリテーター自身の発言は一切ユーザーに表示しない。

---

## 1. 全体構成

```
User → チームセッション（会議目的 = セッションタイトル）
         ↓
       Backend: type=="team" 検出
         ↓
       ─── ファシリテーターループ ───────────────────────────────
       │                                                        │
       │  LM Studio に「会議目的 + 会話履歴 + メンバー情報」を渡す  │
       │  出力: {"next": "agent_001"} or {"next": null}        │
       │                                                        │
       │  next != null                                          │
       │    → ask_agent(agent_id, 会話履歴付きメッセージ)        │
       │    → 回答を SSE ストリーミング表示                       │
       │    → ループ先頭に戻る                                   │
       │                                                        │
       │  next == null → ループ終了                              │
       ─────────────────────────────────────────────────────────
         ↓
       ユーザーに返却・反応待ち
```

---

## 2. データモデル

### agents.json（チームエントリー）

```json
{
  "id": "team_20260407_xxxxxx",
  "name": "コードレビューチーム",
  "description": "コードレビューを行うチーム",
  "type": "team",
  "members": ["agent_001", "agent_002"]
}
```

通常エージェントに `type` と `members` を追加する。`type` が未指定の場合は従来通り `"agent"` として扱う。

### settings.json 追加フィールド

| フィールド | デフォルト値 | 説明 |
|-----------|------------|------|
| `lmstudio_url` | `http://localhost:1234/v1` | LM Studio API エンドポイント |
| `team_max_turns` | `20` | ファシリテーターループの上限ターン数 |

### チームセッションファイル

保存先: `data/team_sessions/{team_id}/{session_id}.json`

```json
{
  "session_id": "...",
  "title": "PR #123 のコードレビュー",
  "created_at": "2026-04-07T12:00:00",
  "messages": [
    {
      "role": "user",
      "content": "このPRをレビューしてください"
    },
    {
      "role": "agent",
      "agent_id": "agent_001",
      "agent_name": "レビュアーA",
      "content": "..."
    },
    {
      "role": "agent",
      "agent_id": "agent_002",
      "agent_name": "レビュアーB",
      "content": "..."
    }
  ]
}
```

## 3. LM Studio 統合

### 起動管理

1. LM Studio API（`lmstudio_url`）に疎通確認（`GET /v1/models`）
2. 失敗した場合、`lms server start` を実行
3. 応答があるまでポーリング（1秒間隔、最大30秒）
4. タイムアウト時はエラーを返す
5. モデルロードは LM Studio 側の自動ロード設定に委任

### ファシリテーター呼び出し

毎ターン以下を渡す:

- **システムプロンプト**: ファシリテーターの役割説明 + メンバー情報（ID・名前・description）
- **ユーザーメッセージ**: 「会議目的: {title}\n\n会話履歴: {...}\n\n次に発言すべきメンバーを選べ」

出力形式（JSON）:

```json
{"next": "agent_001"}   // 次の発言者
{"next": null}          // 議論終了 or ユーザーへ返却
```

---

## 4. セッション管理

### 会議目的

- セッションの `title` フィールドを会議目的として使用
- 最初のユーザーメッセージから自動設定
- ユーザーが後から編集可能（既存のタイトル編集機能を流用）

### メンバー側セッション

Phase 6 の仕組みを流用し、各メンバーに `via team_xxx` セッションを自動生成する。

---

## 5. SSE イベント

| イベント種別 | タイミング | データ |
|------------|-----------|-------|
| `routing` | ファシリテーターが次の発言者を決定 | `"agent_001 が回答中..."` |
| `chunk` | メンバーの回答ストリーミング中 | 既存と同じ |
| `done` | ループ終了 | 既存と同じ |
| `error` | エラー発生 | 既存と同じ |

---

## 6. UI

### チーム作成画面

- エージェント作成フォームに「種別」選択を追加（通常 / チーム）
- チーム選択時、メンバー選択 UI を表示（既存エージェント一覧からチェックボックス選択）
- `path` フィールドは不要（チームには作業ディレクトリがない）

### チームセッション表示

- 発言者をメンバー名で区別して表示（`[レビュアーA]` のようなヘッダー）
- セッション読み込みは `data/team_sessions/` から行い、既存の Claude Code JSONL 読み込みと共存
